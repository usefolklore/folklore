# Deny-Gate Calibration on the REAL Graph

**Status:** evidence-based recommendation, backed only by real-query
measurement. No `src/` threshold was changed by this work.

**TL;DR.** On the real ~21k-node `~/.folklore` graph, the deny-on-confidence
gate **cannot be made to work by retuning `(threshold, minHits)` alone.** The
satisfaction score does not separate in-corpus from out-of-corpus queries
(separability AUC = **0.52**, a coin flip), and the distance the relevance cap
filters on is a synthetic rank-distance that filters out nothing. The shipped
gate fires **0%** of the time on real in-corpus queries at every swept
threshold; the proposed score-only relaxation can fire, but only by accepting a
**45% false-deny rate** to reach 33% true-deny, and the only zero-false-deny
operating point yields just **17% true-deny**. The fix is not a number — it is a
relevance-bearing signal the gate currently does not have.

---

## 1. What was measured, and how

### The harness (calls the real engine, no stubs)

`bench/bench-deny-real.mjs` runs every labeled query through the **real**
application ask path:

```
node dist/cli/index.js ask "<query>" --json --k 5 --workspace all
```

This is the actual use-case — embed → hybrid vector search → cross/PPR/recency
rerank → `computeSatisfaction` → `decideContract` — running in-process against
the real graph, real `vectors.db`, and the real cached embedder under
`~/.folklore/models` (offline, no network). `dist/cli/index.js` is invoked
directly rather than `bin/folklore.js` so the daemon-socket shim is bypassed and
the ask runs in-process. `--workspace all` is **required**: the bench's cwd is
the akashik repo, and without it the CLI applies a workspace pre-filter that
scopes out every other repo's nodes, hiding most of the graph. The harness
captures, per query, the real `satisfaction`, `decision`, and the `hits[]`
distances; it then replays the deny rule over a grid against those cached
probes. Zero probe failures in the reported run.

### The labeled set (reproducible, auditable provenance)

`eval/fixtures/deny-real/` (generated by `generate-fixtures.mjs`, deterministic
seed, re-run after any graph change):

- **36 in-corpus queries** (`expect_deny=true`). Natural recall questions
  synthesised from the **titles of real nodes** that demonstrably exist in
  `graph.json` — 18 arxiv paper titles and 18 indexed web-page titles. Each row
  records `from_node_id`, `from_label`, and `source_uri` as provenance, so any
  reader can open the node and confirm the corpus genuinely contains the answer.
  The label is grounded in graph state; the satisfaction is **measured**, never
  assumed.
- **22 out-of-corpus queries** (`expect_deny=false`). Topically-absent domains
  for a software-research / AI-papers / coding-session corpus — cooking,
  gardening, sports results, world geography, first aid, live market prices,
  classic literature. Each row records the `reason` for asserting absence; the
  bench independently confirms absence via the nearest-hit distance.

**Limitation, stated up front.** This is 36 + 22 queries on **one** real graph.
The rates are a direction, not a population estimate — one query flipping moves
a rate by ~3 points. The conclusion below is robust to that noise *because the
core finding is a near-zero separability AUC, not a marginal rate difference.*

## 2. The real-query satisfaction distribution

The score every threshold must sit against. Histogram (bucket = lower bound),
`#` = in-corpus, `.` = out-of-corpus:

```
sat    in  out
0.15    1    0   #
0.20    1    0   #
0.25    3    2   ###  ..
0.30    6    8   ######  ........
0.35   13    2   #############  ..
0.40    1    7   #  .......
0.45    5    3   #####  ...
0.50    3    0   ###
0.55    3    0   ###
```

| population     | n  | min  | p25  | median | p75  | p90  | max  | mean  |
|----------------|----|------|------|--------|------|------|------|-------|
| in-corpus      | 36 | 0.17 | 0.31 | **0.37** | 0.45 | 0.54 | 0.57 | 0.386 |
| out-of-corpus  | 22 | 0.27 | 0.33 | **0.38** | 0.41 | 0.45 | 0.45 | 0.372 |

**The two distributions are on top of each other.** In-corpus median 0.37,
out-of-corpus median 0.38. Separability **AUC = P(in > out) + ½P(in = out) =
0.52** — statistically indistinguishable from 0.50 (no discrimination). 12 of 22
out-of-corpus queries score at or above the in-corpus median.

The only real signal is in the **upper tail**: 6 in-corpus queries reach
satisfaction ≥ 0.50, and **0** out-of-corpus queries do. That thin tail is the
entire separation available, and it caps true-deny at 6/36 = 17%.

Critically, **the maximum in-corpus satisfaction observed is 0.57** — far below
the `CONTRACT_THRESHOLDS.use_memory = 0.85` breakpoint. No real in-corpus query
gets anywhere near the bar that the shipped gate requires.

## 3. Why the distance relevance-cap is inert

The deny hook's relevance pre-filter keeps only hits with `distance <=
FOLKLORE_HIT_THRESHOLD` (default 1.05; `folklore-smart-hook.cjs:297`). On the
real graph this filter removes **nothing**:

```
surviving_hits (after the d<=1.05 filter):  in-corpus {5: 36}   out-of-corpus {5: 22}
```

Every query — in- and out-of-corpus alike — keeps all 5 hits. `minHits` is
therefore meaningless on the real graph: it is always satisfied.

The reason is structural. The `distance` field that `ask --json` emits is **not
a raw cosine distance**. The ask pipeline rewrites it:

- `multiRrfFuse` (multi-term queries) sets `distance = 1/(1+score)` to encode
  RRF rank (`src/domain/vectors.ts`).
- `pprRerank` (always, single-term too) sets `distance = 1 − fused`, where
  `fused` blends vector similarity with PageRank-style **graph centrality**
  (`src/domain/graph-rerank.ts`).

So a heavily-linked, high-centrality node (e.g. folklore's own protocol docs)
lands near `distance 0` even for a totally off-topic query. Concretely, "how do
I bake sourdough bread" returns folklore's `PROTOCOL-QUALITY-QUESTIONS.md` and
`README.md` chunks at synthetic distance 0.00–0.21. The synthetic top-hit
distance is **0 for 18 of 22 out-of-corpus queries** — the relevance cap waves
them straight through.

Looking at the best *non-zero* distance per query (the closest thing to a real
proximity signal) confirms the cap can't help either: out-of-corpus best-real
distance has a median of ~0.23 — *closer* than the in-corpus median of ~1.0 —
because out-of-corpus queries collapse onto a few high-centrality hubs while
in-corpus queries spread across many specific nodes. The distance signal is, if
anything, **inverted** relative to what a relevance cap needs.

## 4. The swept grid

`threshold ∈ {0.40 … 0.85}` × `minHits ∈ {1, 2, 3}`. (`minHits` has no effect —
all queries retain 5 surviving hits — so only the threshold axis is shown; the
full grid is in `eval/out/deny-real-summary.json`.)

### Variant A — shipped gate (`decision === 'use_memory'` required)

| threshold | true-deny (in) | false-deny (out) |
|-----------|----------------|-------------------|
| 0.40–0.85 | **0 / 36 = 0%** | 0 / 22 = 0% |

The shipped gate is **completely inert on the real graph at every threshold.**
No in-corpus query reaches `decision = use_memory` (max satisfaction 0.57 < 0.85
breakpoint, and `decideContract` also demotes shallow-evidence local hits). The
decision breakdown is `in-corpus: {search_required: 12, ask_user: 24}`,
`out-of-corpus: {ask_user: 12, search_required: 10}`. Turning
`FOLKLORE_DENY_THRESHOLD` does nothing, because the `action === 'use_memory'`
precondition gates the rule shut first.

### Variant B — score-only gate (drops the `use_memory` precondition)

| threshold | true-deny (in)   | false-deny (out)  |
|-----------|------------------|--------------------|
| 0.40      | 12 / 36 = **33%** | 10 / 22 = **45%** |
| 0.45      | 11 / 36 = 31%    | 3 / 22 = 14%      |
| **0.50**  | 6 / 36 = **17%** | **0 / 22 = 0%**   |
| 0.55      | 3 / 36 = 8%      | 0 / 22 = 0%       |
| ≥ 0.60    | 0 / 36 = 0%      | 0 / 22 = 0%       |

This is the whole story in one table. The moment the threshold drops low enough
to fire on real in-corpus queries (≤ 0.45), it *also* fires on out-of-corpus
queries at a punishing rate (14–45%). The only threshold that holds false-deny
at zero is 0.50, and there it denies just 17% of the queries memory could
actually answer.

**Pareto-best operating point (the recommendation, if one must be picked):
score-only gate, threshold 0.50, minHits 1 (minHits is inert).**
True-deny 17%, false-deny 0%.

## 5. Recommendation

### On `CONTRACT_THRESHOLDS.use_memory`

**Do not lower it to chase deny coverage.** Lowering `use_memory` from 0.85
toward the observed real-query band (0.37–0.57) would make the gate fire, but —
as Variant B proves directly, since score-only *is* the use_memory threshold
with its precondition removed — any threshold low enough to catch in-corpus
queries catches out-of-corpus queries too. The current 0.85 is defensible as a
*conservative* setting precisely because the score below it is not trustworthy
as a relevance signal. Keep `use_memory = 0.85`.

### On the deny gate as currently designed

**Good separation is not achievable by recalibration on the real graph.** No
swept cell reaches the "≥ 50% true-deny AND 0% false-deny" bar. The honest
operating point is conservative — score-only @ 0.50 → 17% true-deny / 0%
false-deny — which deflects a sixth of answerable queries while never wrongly
blocking the web. If the priority is "never false-deny," that cell is shippable;
if the priority is "meaningfully reduce web trips," it is not enough, and no
threshold makes it enough.

### The actual fix (a signal, not a number)

The gate has no relevance-bearing input at the decision boundary. Two changes
are required before a deny gate can separate cleanly, and they live in `src/`
(owned by the main thread — flagged here, not changed):

1. **Surface a true cosine/proximity distance alongside the synthetic
   rank-distance.** The relevance cap must filter on real embedding proximity,
   not on `1 − pprFused`. As long as graph centrality dominates the emitted
   `distance`, off-topic hubs will always survive the cap. (The relevance-gate
   work already in `peer-telemetry.ts` — `D_NEAR=0.5`, `D_FAR=1.2`,
   `COVERAGE_WEIGHT=0.6` — uses `bestDistance` internally, but it is fed the same
   synthetic distance, so it cannot help; it needs a real one.)

2. **Let the relevance gate actually move the score on real data.** With the
   in/out satisfaction medians identical at 0.37, the trust aggregate is doing
   all the work and the relevance gate is contributing nothing observable. Once
   it is fed a real proximity (and ideally lexical query-term coverage), the
   in-corpus tail should lift and the out-of-corpus body should fall — *then*
   re-run this exact harness and re-sweep. The bench is built to be re-run.

Until one of those lands, the deny-on-confidence gate should stay
**conservative** (the 0.85 use_memory breakpoint, shipped variant), accepting
that it rarely fires on this graph rather than firing wrongly. That is the
correct failure mode for a feature whose costly error is blocking a web search
the agent actually needed.

## 6. Reproduce

```bash
npm run build
node eval/fixtures/deny-real/generate-fixtures.mjs   # regenerate labels from the live graph
node bench/bench-deny-real.mjs                        # full sweep against the real ask path
# JSON detail (per-query probes, full grid): eval/out/deny-real-summary.json
```

The harness exits 1 with a precise message if the build is missing, the embedder
cache is absent, or there is no `graph.json` — it never fabricates a number.

### Honest caveats

- **Set size.** 36 + 22 queries, one graph. The headline (AUC ≈ 0.52, shipped
  gate 0% true-deny) is far enough from any decision boundary that small-sample
  noise does not change the conclusion, but the *exact* rates (17%, 33%, 45%)
  should be read as ±~3 points.
- **In-corpus labels are recall-shaped.** They ask about a titled artifact known
  to be indexed; this is the friendliest case for the gate. Real agent prompts
  are often broader, which would only *lower* in-corpus satisfaction further —
  the measured separation is an optimistic ceiling, not a floor.
- **`distance` semantics.** The synthetic rank-distance finding is the load-
  bearing diagnosis. It was verified directly: an out-of-corpus query returns
  folklore's own high-centrality docs at synthetic distance 0, while its best
  *cosine-like* non-zero distance is unremarkable. Any future fix must address
  this before re-measuring.
```
