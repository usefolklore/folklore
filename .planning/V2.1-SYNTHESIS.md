## 1. Where all 9 agents agree

Stripping out the persona differences, **every audit pointed at one of four root issues**:

### 1.1 Production code ≠ benchmark code

"The bench script is a parallel universe" — system architect R1. "CI runs `build` + `test` but not the bench" — senior data engineer. "MLOps maturity 3/90" — senior data scientist. "313/313 tests prove the code compiles, not that retrieval got better" — senior data scientist.

Three different agents, three different framings, same fact: **the pipeline that ships is not the pipeline that is measured.** Phase 23 (committed `36c1640`) fixed this for TypeScript by porting the hybrid pipeline into `src/infrastructure/vector-index.ts` and adding `searchHybrid()` to the `VectorIndex` port. But this only works if the encoder is correctly loaded — which §1.2 shows was not true for bge-base.

### 1.2 Encoder weight quality matters more than pipeline optimization

The Phase 22 encoder-swap experiment to `Xenova/bge-base-en-v1.5` gave 63.29% NDCG@10 on SciFact — 11 points below the published 74.04%. Fixing pooling (mean→cls), disabling int8 quantization, expanding max_length — **none of these moved the TS number.**

Tonight's Rust comparison with `fastembed-rs` (which downloads Qdrant-curated ONNX weights, not Xenova's) produced **74.70% NDCG@10, matching the published ceiling within 0.66 points**. Same dataset, same encoder name, same metric code — just a different ONNX conversion.

**The ~11-point gap is an Xenova ONNX port defect, not a pipeline bug.**

Scope of the defect:
- `Xenova/bge-base-en-v1.5`: **-11.4 NDCG** (confirmed defective tonight)
- `Xenova/nomic-embed-text-v1.5`: -0.4 NDCG on SciFact (correct port, within noise)
- `Xenova/all-MiniLM-L6-v2`: matches reference (correct port)

Only `bge-base` is broken. `nomic` and `MiniLM` are fine. But we have **no mechanism to distinguish a correct Xenova port from a defective one** before we've already measured it.

### 1.3 Architectural debt is quantified and deletable

Two independent agents measured it:

**Enterprise architect** — ~2,000 LOC deletable:
- Custom `/folklore/share/1.0.0` libp2p protocol → replaceable with `y-libp2p-provider` standard library
- Hand-rolled tree-sitter parser → replaceable with SCIP ingest (Sourcegraph standard)
- Daemon is a 120-line composition root — extract `p2p-bootstrap.ts`, route CLI peer/share commands through daemon not parallel to it

**Senior data engineer** — `graph.json` is an exponential I/O amplifier:
- Every chunk ingest: 5 full graph loads + 4 full rewrites
- On a 10k-node graph, a 2KB update becomes ~1GB of I/O
- `code-graph.db` already uses SQLite correctly; the research graph never got migrated
- At ~10k nodes this breaks, and we're currently at 9,878

Combined, these findings say: **the TypeScript codebase has ~3,000 LOC of deletable/replaceable work that would improve correctness, simplify ops, and unlock scale — without any feature loss.**

### 1.4 Rooms/tunnels are a latent geometric asset, currently cosmetic

The mathematician, the S2 geometry skill, the data researcher, and the enterprise architect all said some version of: **rooms are structurally meaningful but the flat Euclidean retrieval path doesn't exploit them.**

- **Data researcher** disproved my Wave 4 "null result" on rooms — it was Simpson's paradox. Rooms help +1.0 NDCG on hard sub-populations (mathematica) but the aggregate hid the win because the easy subforum (gaming, 55% of queries) drowned it out.
- **Mathematician** (Proposal B): `findTunnels` is O(n²) brute force today. Replace with Relative Neighborhood Graph via sparsified kNN → O(n log n). Tunnels become a *structural invariant* (cross-color Delaunay edges) not a threshold heuristic.
- **Mathematician** (Proposal A): Hyperbolic embeddings (Poincaré ball / Lorentz, 32-64 dim) for the room hierarchy. Score = λ·hyperbolic_prior + (1-λ)·cosine. Makes rooms a **continuous structural prior** instead of a hard SQL filter. +1.5 to +3 NDCG expected.
- **S2 geometry** (discrete version of the same idea): 64-bit cell IDs + B-tree range scans + cell-addressed rooms replaces `sqlite-vec`'s fragile `vec0` with a plain SQLite B-tree. Rooms become integer hierarchy. findTunnels speedup 26-316× depending on corpus size.

The S2 and hyperbolic proposals are **complementary**: S2 is the integer/indexing version; hyperbolic is the continuous/scoring version. Both agree rooms deserve a first-class role in retrieval.

## 2. What the Rust measurement changed

Tonight's `folklore-rs/` crate tested two things in parallel:

### 2.1 TS vs Rust on identical weights — tied

SciFact MiniLM, both runtimes driving the exact same Xenova ONNX file:

| | NDCG@10 | MRR | Indexing |
|---|---|---|---|
| TS (@xenova/transformers) | 64.82% | 0.604 | 19 docs/sec |
| Rust (fastembed) | 65.27% | 0.611 | 19 docs/sec |

**Quality tied within noise (+0.45 NDCG). Throughput IDENTICAL.** Both runtimes bottleneck on the same native ONNX Runtime — the language overhead is negligible. Rewriting folklore in Rust is NOT an inference speedup.

### 2.2 TS vs Rust on a defective Xenova port — Rust wins by 11 NDCG

SciFact bge-base, Rust uses Qdrant-curated ONNX, TS uses Xenova-curated:

| | NDCG@10 | Recall@10 | Matches published? |
|---|---|---|---|
| TS Xenova | 63.29% | 75.69% | -11 (DEFECTIVE) |
| Rust fastembed | **74.70%** | **86.69%** | +0.66 (MATCH) |
| Published BAAI | 74.04% | 87.42% | — |

**This is not a language advantage. It is a weight-selection advantage.** fastembed downloads Qdrant-published ONNX conversions that are quality-gated. Xenova's library auto-downloads whatever is in the `Xenova/` HF namespace, with no validation.

### 2.3 What this means for the v2.1 plan

Two truths that reshape everything:

1. **The "Rust rewrite for speed" thesis is wrong.** Both runtimes are inference-bound on ONNX Runtime. Rust wins nothing on throughput when weights are equivalent.

2. **The "TS stack is fine" thesis is also wrong.** We cannot trust any `Xenova/*` port without validating it — and we have no validation mechanism. Nomic is fine, bge-base is broken, we don't know about anything else we might swap in.

The Rust crate's real value is as a **weight-quality ground truth**: run the same BEIR benchmark through a completely independent ONNX conversion and compare. If Rust and TS agree, Xenova is fine. If they disagree, Xenova is broken.

## 3. Three paths forward

Given the consolidated findings and the Rust measurement, there are three realistic shapes v2.1 could take.

### Path A: Rust inference sidecar (pragmatic, 3-5 days)

**Shape:** Keep TypeScript folklore as-is for MCP server, CLI, daemon, P2P, codebase indexing, session persistence. Add a tiny Rust bin (`folklore-embed`) that reads texts from stdin and writes vectors to stdout. Replace the current `xenovaEmbedder` in `src/infrastructure/embedders.ts` with a new `rustSubprocessEmbedder` that spawns the Rust bin and streams through it.

**What ships:**
- New Cargo bin in `folklore-rs/src/bin/embed-server.rs` — stdio JSON protocol, one request = one vector or batch
- New TypeScript adapter in `src/infrastructure/embedders.ts` — spawns Rust child process on first use, caches the handle, streams batches
- Production folklore automatically uses fastembed-quality weights for bge-base
- nomic still works via either route (correct ports)
- Zero changes to MCP/CLI/daemon/P2P — pure inference-layer swap

**Pros:**
- Smallest scope that captures the proven weight-quality win
- 3-5 days of work, low architectural risk
- Users don't see any difference in UX or deployment
- Rust crate stays tiny — just the bench + embed server

**Cons:**
- Subprocess IPC overhead per batch (~1-5ms fixed cost) — fine for production-scale, dominant for tiny benchmarks
- Doesn't address any of the TS-side architectural debt (graph.json, custom share protocol, tree-sitter parser)
- Doesn't unlock any of the room/geometry proposals
- Two binaries to deploy instead of one

**Estimated effort:** 3-5 days, ~400 LOC new Rust, ~200 LOC new TS adapter

### Path B: Hybrid Rust/TS (balanced, 4-6 weeks)

**Shape:** folklore-rs becomes the retrieval subsystem — inference, vector index, BM25, FTS5, tunnel detection, room routing. TypeScript folklore keeps MCP server, CLI, daemon, P2P, codebase indexing, sessions, source adapters. The two talk via a stdio JSON protocol (or Unix socket) where Rust is the server and TS is the client.

**What ships:**
- Cargo workspace: `folklore-rs/` gains `core/`, `bench/`, `server/` sub-crates
- `core` has the retrieval primitives (encoder port, hybrid pipeline, RRF, tunnels, room routing)
- `server` exposes them via stdio (`embed`, `search_global`, `search_room`, `upsert`, `tunnels`, `room_route`)
- TypeScript `VectorIndex` gets a new adapter: `rustRetrievalIndex` that speaks to the Rust server
- Existing `sqliteVectorIndex` stays as a fallback for unit tests
- Production research graph storage stays in TypeScript (`graph-repository.ts`) — but vectors migrate to Rust-managed storage

**What gets killed:**
- Xenova embedder dependency
- Most of `src/infrastructure/vector-index.ts` (delegated to Rust)
- `scripts/bench-beir-sota.mjs` (replaced by `folklore-rs` bench binary with the same output schema)

**What stays in TS:**
- MCP server (16 tools)
- CLI (ask, room, peer, share, codebase, session commands)
- Daemon (tick loop, libp2p node lifecycle, room sharing)
- Y.js CRDT room sharing
- Tree-sitter codebase indexer (separate aggregate, not on retrieval hot path)
- Source adapters (arxiv, hn, rss, etc.)

**Pros:**
- Retrieval quality matches published (because we use Qdrant/BAAI ONNX ports)
- Retrieval throughput matches Rust native (no subprocess per query — stream over long-lived socket)
- CI regression bar becomes possible (Rust bench is deterministic, testable)
- TS-side surface area shrinks by maybe 30-40% (delete vector-index.ts + bench scripts + reranker code)
- Rust-side stays tight — just retrieval primitives, not a full rewrite
- Gradual — each phase is independently shippable

**Cons:**
- Two-language architecture to maintain
- IPC protocol is a new interface to version
- Requires users to have a Rust binary alongside the Node package (can bundle via npm postinstall or prebuilt binary per-platform)
- 4-6 weeks of work, not 3-5 days

**Estimated effort:** 4-6 weeks, ~2,500 LOC Rust, ~500 LOC TS delete, ~800 LOC TS add

**Phase breakdown:**
- Phase 24 (3 days): Rust server binary — stdio JSON protocol with `embed` + `batch_embed`
- Phase 25 (4 days): TS adapter `rustRetrievalIndex`, replaces dense stage of existing `searchHybrid`
- Phase 26 (3 days): BM25 + hybrid RRF moves to Rust (delete the duplicate domain functions in TS)
- Phase 27 (3 days): Tunnel detection moves to Rust (implements RNG graph per mathematician Proposal B)
- Phase 28 (4 days): Room routing moves to Rust (pilot-centroid classifier per AI solution architect recommendation, or hyperbolic prior per mathematician Proposal A — gate-tested)
- Phase 29 (3 days): CI regression bar on the Rust bench (`cargo test --release` includes scifact-mini with NDCG@10 ≥ 0.72 assertion)
- Phase 30 (4 days): Delete tree-sitter parser + custom share protocol per enterprise architect (optional, can defer to v2.2)

### Path C: Full Rust folklore v3.0 (ambitious, 2-3 months)

**Shape:** folklore-rs grows into a complete replacement. All 16 MCP tools, CLI, daemon, P2P, codebase indexing, sessions, source adapters — all in Rust. The TypeScript codebase enters maintenance mode and eventually gets archived.

**What ships:**
- Full Cargo workspace with ~10 sub-crates
- MCP server in Rust (using one of the Rust MCP libraries or a hand-rolled stdio JSON-RPC handler)
- libp2p-rs (already mature) for the P2P layer
- rusqlite for storage (replaces graph.json AND code-graph.db AND vectors.db in one move)
- tree-sitter-rust bindings for codebase parsing (no change from current TS — tree-sitter is a C library either way)
- reqwest for HTTP sources (arxiv, hn, rss, etc.)
- Single binary deployment — no Node dependency at all

**Pros:**
- Single language, single binary, single dependency tree
- Deletes the ~2,000 LOC of TS tech-debt the enterprise architect identified (because it's just not in scope for the rewrite)
- Fastest possible retrieval + fastest possible ingest
- Production-ready monitoring (Prometheus metrics, structured logging)
- All the data engineer's pipeline recommendations land natively (retry, backpressure, observability)

**Cons:**
- 2-3 months of work
- Risks: libp2p-rs vs libp2p-js compatibility (the existing TS peers wouldn't interop with Rust peers unless the protocol is bit-compatible)
- MCP library ecosystem in Rust is thinner than in TS — might need hand-rolling
- Y.js CRDT in Rust is possible but less mature than y-js
- Forks the user base during the transition

**Estimated effort:** 10-12 weeks, ~10,000 LOC Rust, ~all-of-TS delete

## 4. Recommendation

**Path B (Hybrid Rust/TS).** Three reasons:

1. **Captures the proven win.** The bge-base finding means Xenova cannot be trusted for retrieval quality, full stop. Anything that ships bge-base through Xenova is shipping a -11 NDCG defect. Path A (inference sidecar) is the minimum fix, but Path B gets the whole retrieval hot path into the clean Rust implementation, not just the encoder.

2. **Addresses ~70% of the audit findings without full rewrite.** Path B deletes the duplicate bench pipeline, the flawed vector-index adapter, the dead reranker code, and unlocks the mathematician's RNG tunnel proposal + room routing. It leaves intact the parts the enterprise architect said were cost-not-moat (custom share protocol, tree-sitter parser) — those can be deleted in v2.2 when we have bandwidth.

3. **Shippable per phase.** Path A is "all or nothing in 5 days." Path C is "all or nothing in 10 weeks." Path B is "ship a phase every week, measure the delta, keep or revert per phase." Matches the user's rejection of "piling models" — every phase has an explicit gate test.

**Path B lets us keep the Rust crate's clean DDD architecture as the blueprint for retrieval, while the TS stack keeps handling the parts Rust doesn't improve** (MCP protocol, libp2p, daemon, tree-sitter, Y.js CRDT).

**Not recommended: Path C right now.** The full rewrite is the wrong scope for a single milestone. Most of the TS code (MCP tools, daemon loop, source adapters, CRDT sharing) is correct and working — rewriting it in Rust would be expensive busywork with no user-visible win. Better: let Path B prove the Rust subsystem at scale, then decide in 2-3 milestones whether to continue the migration.

## 5. v2.1 phase breakdown (Path B)

**Milestone:** v2.1 "Rust Retrieval Subsystem + Debt Reduction"
**Estimated:** 7 phases, 4-6 weeks of work
**Gate per phase:** paired-bootstrap significance test on the BEIR-8 subset — no merge without p < 0.05 quality gain or parity

### Phase 24 — Rust embed server (3 days)

**Goal:** ship a standalone Rust binary that embeds text batches via stdio JSON. Replaces `xenovaEmbedder` for the bge-base case (and optionally nomic, though nomic's Xenova port is fine).

**Protocol:** one request = `{ "model": "bge-base", "texts": [...], "is_query": bool }`. One response = `{ "vectors": [[...], [...]] }`. Length-prefixed framing over stdio.

**Deliverables:**
- `folklore-rs/src/bin/embed_server.rs`
- `src/infrastructure/embedders.ts` — new `rustSubprocessEmbedder(model, dim)` adapter that spawns and pools the binary
- End-to-end TS test: `indexNode` and `searchGlobal` work with the Rust embedder
- Documentation: how to build the Rust binary and where the TS adapter finds it

**Gate:** SciFact NDCG@10 via production TS `searchHybrid` using Rust embedder matches the Rust-native bench within 0.5 points (74% ± 0.5).

### Phase 25 — Rust retrieval server (4 days)

**Goal:** move the hybrid retrieval path itself into Rust. Rust server now accepts `search_hybrid` and `search_by_room_hybrid` requests and returns ranked matches.

**Deliverables:**
- Rust server gains `search_hybrid` op (dense via fastembed + BM25 via tantivy or rusqlite FTS + RRF fusion)
- TS `VectorIndex` gains `rustRetrievalIndex` adapter
- Existing `sqliteVectorIndex` kept for fallback
- Benchmark the end-to-end TS → Rust → back path on SciFact/ArguAna

**Gate:** end-to-end NDCG@10 ≥ 74% on SciFact, latency p50 ≤ 80ms (2x Rust-native baseline allows for IPC overhead).

### Phase 26 — BM25 + FTS unification (3 days)

**Goal:** eliminate the duplicate BM25 path. Everything sparse goes through the Rust server.

**Deliverables:**
- Delete `fts_docs` table creation + BM25 prepared statements from TS `vector-index.ts`
- Rust server owns the sparse index (either `tantivy` or `rusqlite` FTS5 — decide based on deployment simplicity)
- TS `searchHybrid` becomes a ~10-line wrapper around the Rust RPC call
- Keep the domain-layer `sanitizeForFts5` + `rrfFuse` pure functions in TS for tests, but the production path is Rust

**Gate:** 313/313 TS tests still pass. BM25 latency on ArguAna (long queries) p50 ≤ 100ms.

### Phase 27 — Tunnel detection as RNG graph (3 days)

**Goal:** implement mathematician Proposal B (Voronoi/RNG foam tunnels) in Rust. Replaces O(n²) `findTunnels` with O(n log n) RNG sparsified from kNN.

**Deliverables:**
- `folklore-rs/src/domain/tunnel_graph.rs` — pure RNG construction from an approximate kNN graph
- Rust server gains `compute_tunnels` + `query_tunnels` ops
- TS `findTunnels` use case delegates to Rust
- Offline tunnel cache per system architect R8 — tunnels computed at ingest time, not per-query

**Gate:** tunnel computation on a 10K-node corpus completes in under 1 second (vs current O(n²) which is ~minutes). Tunnel quality (manual spot-check on 10 known cross-room pairs) is equal or better.

### Phase 28 — Room routing (3-5 days)

**Goal:** make rooms a first-class retrieval signal via the AI solution architect's recommended query-type classifier pattern, OR the mathematician's hyperbolic prior — gate-tested against each other.

**Deliverables:**
- Rust server gains `route_query` op — returns a scored list of rooms the query might belong to
- Two implementation variants to compare:
  - **Variant A**: pilot-centroid routing per RouterRetriever (AAAI 2025) — compute room centroid embeddings at ingest, route query by cosine-to-centroid at query time
  - **Variant B**: hyperbolic prior (mathematician Proposal A) — 32-dim Poincaré embedding of the room tree, score = λ·hyperbolic_prior + (1-λ)·cosine
- TS `searchByRoom` use case delegates to Rust
- Paired-bootstrap significance test on a multi-room benchmark (CQADupStack 3-subforum or synthetic mini-LoTTE) — whichever variant wins by ≥1 NDCG at p < 0.05 ships; if neither, rooms stay a filter

**Gate:** at least one variant produces +1 NDCG on the multi-room benchmark. If both null → accept that rooms are not a retrieval signal in this corpus and document it.

### Phase 29 — CI retrieval regression bar (2 days)

**Goal:** `cargo test --release` in `folklore-rs/` runs a mini SciFact bench (200 passages × 20 queries) and asserts NDCG@10 ≥ 0.72. Plus matching TS-side test that runs the end-to-end path through the Rust server. Per senior data scientist audit.

**Deliverables:**
- `folklore-rs/tests/scifact_mini.rs` — real ONNX, real 200-passage fixture, assertion
- `tests/phase29.rust-retrieval.test.ts` — spawns Rust server, runs the same fixture through TS adapter, asserts NDCG@10 within 0.5 of Rust-native
- GitHub Actions workflow that runs both on every PR and blocks merge on regression
- Data scientist wrote the specific yaml for this already in their audit

**Gate:** CI passes. First real merge that tries to regress retrieval gets blocked by this.

### Phase 30 — Debt reduction (optional, 4 days)

**Goal:** delete the ~2,000 LOC of enterprise-architect-identified tech-first code.

**Deliverables:**
- Delete `src/telegram/` (4 files — not in moat)
- Delete decorative CLIs per architect audit
- Delete reranker branch from bench-beir-sota.mjs (Wave 3 dead)
- Migrate `graph.json` to SQLite per senior data engineer recommendation
- Optional: evaluate `y-libp2p-provider` as replacement for `/folklore/share/1.0.0`

**Gate:** 313/313 still pass. Bundle size / LOC metric drops by ≥20%.

## 7. Open questions for the user

Before kicking off Phase 24, I need your decisions on:

1. **Path B vs Path A?** Path A is 3-5 days and captures the immediate quality win. Path B is 4-6 weeks and captures the audit findings. My recommendation is B.

2. **Which variant for Phase 28 room routing?** Pilot-centroid (simple, proven by RouterRetriever paper) or hyperbolic (novel, higher ceiling per mathematician proposal)? Or both and gate-test?

3. **Bundle the Rust binary or require external install?** Options:
   - npm postinstall downloads a prebuilt Rust binary for the current platform
   - Require users to `cargo install folklore-rs` alongside `npm install folklore`
   - Ship as a single Cargo workspace and let users build both from source

4. **IPC transport?** stdio JSON-RPC (simplest) vs Unix socket (faster) vs gRPC (most complex but standard). My recommendation: stdio for v2.1, migrate to Unix socket in v2.2 if benchmarks show it matters.

5. **Deletion of tech-first code** (Phase 30): pull the trigger now or defer to a v2.2 "debt reduction" milestone? My recommendation: defer Phase 30, keep v2.1 focused on retrieval quality.
