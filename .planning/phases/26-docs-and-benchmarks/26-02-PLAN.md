<!-- The deny-on-confidence behavior, sourced from the project CLAUDE.md "akashik" section and RFC-0001 "Deny semantics". Use these EXACT defaults. -->

Deny-on-confidence gate (the network-before-web behavior):
  - A PreToolUse hook intercepts outbound WebSearch / WebFetch calls (local Read/Grep/Glob are NOT gated).
  - Before the web call, the hook asks the local graph + connected peers.
  - A deny is permitted IFF ALL THREE hold (from RFC-0001 "Deny semantics", keep consistent):
      * satisfaction_score >= AKASHIK_DENY_THRESHOLD (default 0.85)
      * hit count >= AKASHIK_DENY_MIN_HITS (default 2)
      * the decision layer returns use_memory (not verify / search_web)
  - On deny: the hook injects the graph hits into the caller's context in place of the web result.
  - On any failure / no confident answer: the web call proceeds normally; a PostToolUse hook auto-saves the result so the next session hits the graph.
  - Deny is OFF by default (opt-in): a false deny costs more trust than a redundant fetch. (RFC-0001 open question 3 is live.)
  - Env knobs: AKASHIK_DENY_WEBSEARCH=1 (enable), AKASHIK_DENY_THRESHOLD=0.85, AKASHIK_DENY_MIN_HITS=2, AKASHIK_PREFETCH_PEERS=0 (local-only).
  - Freshness interplay: a hit inside its staleness window is trusted; past it, prefer a fresh pull (research <7d, toolshed <30d).
